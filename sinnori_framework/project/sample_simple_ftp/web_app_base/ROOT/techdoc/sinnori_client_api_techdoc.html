1   클라이언트용 서버 접속 API

2013.06.22 현재 자바용 클라이언트 서버 접속 API 만 지원한다.  자바 클라이언트에서 서버와의 데이터 입출력을 위한 가장 중요한 자원이 바로 소켓 혹은 소켓 채널이다. 자바 1.2 시절  대표적인 클라이언트용 서버 접속 API 의 모습은 소켓 폴이다.  소켓 폴 방법은 소켓 큐 관리자를 두어서 메시지를 보내고자 할 때 관리자에게서 소켓을 얻고 소켓을 통해 메시지를 쓰고 읽는 작업을 수행 후 관리자에게 소켓을 반환하는 방법을 말한다.  신놀이 프레임 워크에서는 큰 구도에서는 이 구조를 그대로 따르지만 jdk 1.4 이후 나온 비동기 API 를 활용한 방법을 제공한다.

1.1   용어 정의

1.1.1   연결 클래스

소켓 채널을 감싸아 소켓 채널관련 서비스를 구현하는 클래스, 즉 소켓 채널 랩 클래스를 말한다. 참고) 이렇게 랩을 하는 이유는 큐에 이중 등록이라든지 개발자 실수로 반환을 하지 않는등에 대한 대처를 하기 위함이다. 

1.1.2   비동기 연결 클래스

소켓 채널의 blocking-mode 를 비동기로 설정한 연결 클래스

1.1.3   동기 연결 클래스

소켓 채널의 blocking-mode 를 동기로 설정한 연결 클래스

1.1.4   공유 비동기 연결 클래스 

소중한 소켓 채널을 다수 사용자와 공유하기 위한 목적으로 만들어진 비동기 연결 클래스로 내부적으로 메일함이라는 개념으로 소켓 채널를 공유하면서 메시지 교환을 수행한다.

1.1.5   메일함

비동기 소켓 채널을 다수 사용자가 공유하면서 메시지 교환을 해 주기 위해 만든 개념이다. 메일함은 공유 비동기 연결 클래스에서 메일함 큐로 관리되며 사용자 입장에서는 보면 실질적인 메시지 교환 수행자이다.

1.1.6   비공유 비동기 연결 클래스

오직 1명의 사용자만이 독점해서 사용하기 위한 비동기 연결 클래스

1.1.7   비공유 동기 연결 클래스

오직 1명의 사용자만이 독점에서 사용하기 위한 동기 연결 클래스로 jdk 1.2 시절 소켓 폴과 같다.

6   비공유 비동기 메시지 교환 방법 버전2

7   비동기 불특정 메시지 교환 방법

서버에서 메일함 식별자 0번으로 보내는 메시지로 자신이 보낸 입력 메시지의 결과로 얻은 출력 메시지가 아닌 서버 자체로 보낸 출력 메시지를 말한다.  이것이 필요한 경우의 예를 들면 채팅의 경우 내가 보낸 메시지는 채팅 그룹원들 전체로 보내진다. 채팅 그룹에 속한 그룹원들 입장에서는 자신이 보낸 메시지가 아닌 그룹원이 보낸 메시지를 받게 된다. 이때 받은 메시지를 불특정 메시지라 한다.

7.1    기본 아이디어

비동기 메시지 교환에서 메일 식별자 0인 출력 메시지는  서버 공유 메시지 큐로 보낸다. 서버 공융 메시지 큐를 전담하는 프로세스(=쓰레드)가 있어 이를 처리한다.

9   개선사항 혹은 이슈

9.1   비동기 메시지 교환 방식에서 입력 메시지 소켓 쓰기 담당 쓰레드

9.1.1   삭제시  장점 혹은 유지할 필요가 없는 이유

9.1.1.1   소켓 쓰기 담당 쓰레드와 입력 메시지 큐  자원이 삭제된다.

9.1.1.2   소켓 쓰기 담당 쓰레드는  소켓 동기화를 하면서 쓰기를 하므로 삭제시 동기화 비용이 감소함 .  단 공유 비동기  방식은 이에 해당되지 않는다.   주) 영향력은 측정 못했음.

9.1.1.3   래퍼 메시지를 래퍼 번호순으로 보낼 수 있다.  래퍼 메시지 그룹에 속한 래퍼 메시지 끼리도 소켓 동기화  쓰기 경쟁을 통해 서버에 전달 되기 때문에 이웃하는 래퍼 메시지 순서가 다소 어긋 날 수 있다.  서버 입장에서 번호순으로 온다고 하면 비 순차적으로 오는것은 공격으로 인식할 수 있으며 특히  래퍼 메시지 그룹 생성은 래퍼 번호 1번만으로 제약할 수 있다. 그러면 자원 보호에 좀더 유리한 위치에 서게 된다.

9.1.2   삭제시 단점 혹은 유지해야 할 이유

9.1.2.1   더 많은 쓰기 전용 버퍼를 요구한다. 

소켓 쓰기 담당 쓰레드는 다른 쓰레드와 공유하지 않는 쓰기 전용 버퍼를 가지고 있다.  소켓 쓰기 담당 쓰레드가 가지고 있는 쓰기 전용 버퍼를 연결 클래로 옮겨 와야 하는데 아래 2가지 방법으로 옮겨 올 수 있다.

9.1.2.1.1   쓰기 전용 버퍼를 큐를 이용하는 방법

쓰기 전용 버퍼 큐로 부터 쓰기 버퍼를 얻어 소켓 쓰기 작업 완료 후 쓰기 전용 버퍼 큐로 쓰기 버퍼를 반환하는 방법으로 큐 크기 만큼 전용 버포를 소진한다.  쓰기 전용 버퍼를 쓰레드 세이프한 큐로 관리 하기 때문에 이렇게 얻은 쓰기 전용 버퍼를 이용하면 쓰레드 세이프 하게 데이터 패킷을 꾸릴 수 있다.  단점으로는 쓰기 버퍼 큐 확보하기 위한 동기화 비용이 추가 발생한다.

9.1.2.1.2   동시 사용자 수 만큼 할당 받는 방법

비공유 비동기인 경우 연결 클래스 만큼, 공유 비동기인 경우 메일함 갯수 만큼 즉 동시 사용자 수 만큼 할당 받아 사용하면 쓰레드 세이프하게 데이터 패킷을 꾸릴 수 있다.  단점으로는 많은 수의 쓰기 전용 버퍼가 필요하다는 점이다.

9.1.2.2   데이터 패킷 크기 만큼만 작업 하는 소켓 쓰기 작업은 짧은 시간에 이루어 지는 작업으로 동기화를 유지하는 시간이 짧다.  따라서 이것이 부하의 원인이 되는지는 꼼꼼하게 따지지 않고 삭제하는 것은 옳지 않다.
